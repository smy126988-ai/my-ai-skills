# vibe-brainstorm 需求脑暴助手

## 核心功能

### 1. 预搜索(Pre-search)
技能启动后,在发起 AskUserQuestion 之前,先用 Tavily 搜索用户话题相关的现有方案:

- **搜索策略**:构造 2-3 个关键词组合,如 `"<主题> GitHub"`、 `"<主题> 开源/工具"`、 `"<主题> 教程/方案"`
- **结果筛选**:取前 3-5 条最相关结果,重点关注 GitHub 项目、成熟工具、已有解决方案
- **上下文注入**:将搜索结果整理为简短摘要,作为第一轮 AskUserQuestion 的上下文

**目标**:避免从零开始的"你要做什么"式澄清,直接基于现有方案问关键决策点。

> 示例:用户说"想做个抢票器" → 搜索"智谱 GLM 抢购 GitHub" → 发现 glm-rush → 第一轮问"发现已有 glm-rush 项目,你是想直接用它做适配,还是从头重写?"

### 2. 对话式脑暴
通过 AskUserQuestion 与用户进行多轮对话,把模糊想法变成清晰需求。

**追问策略**:场景→功能→边界→技术
- 每次只问 1-2 个问题
- 第一轮基于预搜索结果直接问关键决策点,不做从零澄清
- 基于用户回答中的模糊点生成追问
- 需求过于宏大时主动提示风险,给出 MVP 替代方案

### 摘要触发规则
- **强制触发**:第 3 轮对话后**必须**生成第一次摘要
- **周期性触发**:之后每 2-3 轮生成一次
- **内容要求**:摘要必须包含三部分——已确认/待确认/当前聚焦
- **状态持久化**:每轮对话后将新确认的信息增量写入 brainstorm_state.json

### 可行性挑战判断清单
出现以下**任一信号**即触发挑战:
- 涉及第三方 API/平台对接(飞书/钉钉/企业微信/Slack 等)
- 需要多系统数据联动(>2 个系统)
- 开发周期估算超过 1.5 周
- 涉及数据安全/合规要求
- 需要用户改变现有工作流(迁移成本)

**挑战方式**:不是否定,而是对比——
> "这个方案要做 X+Y,估计至少需要[时间/资源]。如果先只做[核心子集],能不能先验证[核心假设]?"

**落地**:讨论充分后,调用 save-requirement 写入 .vibe/requirements.md

### 2. 状态管理
- resume - 恢复上次脑暴状态
- clear - 清空当前状态(自动归档旧版本)
- report - 生成结构化进度报告 + Mermaid 可视化图表

### 3. 可视化
- 思维导图:按角色分组的用户场景
- 流程图:场景→功能→完成
- 自动保存到 .vibe/brainstorm_visuals.md

### 4. 行业搜索
- Tavily 搜索 + Claude 知识降级
- 24 小时缓存,避免重复搜索
- 输出格式:"你的需求 vs 行业方案"对比表

## 数据文件(.vibe/)
```
.vibe/
├── requirements.md       # 需求列表
├── brainstorm_state.json # 脑暴状态
├── brainstorm_report.md  # 进度报告
├── brainstorm_visuals.md # Mermaid 图表
├── search_cache.json     # 搜索缓存
└── archives/             # 状态归档
```
